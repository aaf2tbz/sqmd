# sqmd — What It Is

## The Problem

Every AI agent that reads code does it poorly. Here's why:

1. **File-level granularity.** Agents read entire files when they need one function. A 500-line file with 15 functions wastes 93% of the tokens on irrelevant code.

2. **Keyword-only search.** Grep and ripgrep are fast but blind. Searching for "auth" won't find `authenticate_credentials` unless you think to search for it.

3. **No structural awareness.** When an agent finds the auth middleware, it has no idea what imports it, what calls it, or what it depends on.

4. **Expensive extraction pipelines.** Many tools rely on cloud APIs or LLM calls for code understanding — slow, expensive, offline-incapable.

5. **Context assembly is ad hoc.** Agents manually stitch together file reads, grep results, and glob matches.

6. **Cue mismatch.** Developers search with natural language ("how does authentication work") while code uses technical identifiers (`authenticate_credentials`). This semantic gap causes retrieval misses even with good keyword search.

## The Solution

sqmd is a local-first code index that makes codebases fully queryable by agents in under 20ms.

### How It Works

**Ingestion** — tree-sitter parses every source file into semantic chunks: functions, classes, methods, interfaces, types, modules. Each chunk stores raw source code (`content_raw`), not pre-rendered Markdown. Also accepts external knowledge (facts, decisions, preferences) via the ingest API. Batches of knowledge automatically produce session summaries for document-level retrieval.

**Storage** — Every chunk goes into a single SQLite file alongside:
- An FTS5 full-text index with Porter stemming for keyword search
- Vector embeddings (via local ONNX model) for semantic search
- Vector embeddings over hint text for semantic hint retrieval
- An import/call/contains relationship graph for dependency traversal
- An entity knowledge graph with hints for bridging the semantic gap

**Hint generation** — Three types of hints bridge the cue mismatch gap:
1. Template hints (always): "how does authenticate work", "authenticate implementation"
2. Relational hints (always): "X implements Y", "X calls Y" from the entity graph
3. LLM prospective hints (optional): Natural-language queries generated by gemma3:4b via Ollama, predicting what a developer would search for

**Query** — A hybrid search engine combines vector similarity (70%) and keyword relevance (30%), boosted by entity graph density and importance scoring. Hint vector KNN runs in parallel with content vector search, providing a fourth scoring signal. Results are returned as JSON with both structured fields and a pre-rendered `"markdown"` field — agents grab `markdown` directly into their prompts, no extra formatting step.

**Incremental updates** — A file watcher detects changes and re-indexes only what changed, in under 200ms. Content-hash decision pipeline: SKIP (unchanged), UPDATE (modified), TOMBSTONE (deleted).

### Performance Characteristics

| Operation | Latency |
|-----------|---------|
| Full index (10k files, cold) | <60s |
| Incremental re-index (1 file) | <200ms |
| Hybrid search query | <20ms |
| Hybrid search + hint vectors | <25ms |
| Dependency traversal (depth 2) | <50ms |
| Context assembly (8k tokens) | <5ms |
| Idle daemon | ~15MB RAM, 0% CPU |

### Design Philosophy

1. **Local first.** No network, no API keys, no cloud services. A single SQLite file per project. Optional Ollama integration runs locally.

2. **Deterministic over probabilistic.** Use tree-sitter parsing instead of LLM extraction. Use cosine similarity instead of LLM-based deduplication. When LLMs are used (hint generation), outputs are deterministic with fixed prompts.

3. **Raw code, derived Markdown.** `content_raw` stores original source. Markdown derived on demand via `Chunk::render_md()`. Every query response includes both structured fields and a `markdown` field.

4. **Single binary.** Rust + static linking. No runtime dependencies. Drop it on any machine and it works. Optional features add capability without adding runtime requirements (Ollama must be running for hint generation, but the binary works without it).

5. **Agent-native output.** JSON responses with a pre-rendered `markdown` field that agents can inject directly into their prompt context. No parsing, no formatting step.

6. **Research-driven retrieval.** Retrieval architecture shaped by empirical evidence, not assumptions. The Obsidian Vault Recall Eval showed that hybrid retrieval beats lexical-only (+12.5% Hit@1), semantic hint retrieval adds real value (+2.6% Hit@1), and note-level retrieval beats chunked on knowledge. sqmd implements all three findings.

## Intended Use Cases

**Primary:** Agent context retrieval — the system that answers "what code do I need to read right now?" for AI coding assistants.

**Secondary:** Codebase navigation for humans — a CLI that understands code structure better than any grep.

**Tertiary:** Knowledge management — ingest facts, decisions, and session summaries alongside code, with unified search across both.
